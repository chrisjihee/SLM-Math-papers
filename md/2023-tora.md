## 2. One-line Summary

ToRA는 자연어 reasoning과 program/tool 호출·실행 피드백을 한 trajectory 안에서 interleave하는 tool-integrated reasoning agent를, GPT-4 tool-use trajectory에 대한 imitation learning과 output space shaping으로 7B~70B LLaMA/CodeLLaMA에 학습시켜 open-source math reasoning을 크게 끌어올린 논문이다.

## 3. 핵심 문제 설정

- 이 논문이 푸는 질문은 **작은 open model이 복잡한 수학(특히 MATH 수준)을 풀 때, 순수 CoT의 계산·정밀도 한계와 순수 program(PAL/PoT)의 추상적 추론 한계를 어떻게 동시에 넘느냐**이다.
- 기존 한계로 보는 것:
  - rationale-based CoT: 의미적 추론은 좋지만 계산·symbolic 처리에서 부정확하다.
  - program-based PAL/PoT: 계산은 정확하지만 복잡한 의미적 추론(기하·고난도 단계 분해)에는 약하다.
  - 둘은 보통 분리된 단일 path family로 쓰여, 한 문제 안에서 상호보완되지 못한다.
- 해법은 **single interleaved trajectory**: 자연어 reasoning → program 작성 → interpreter 실행 → 실행 결과를 다시 자연어 reasoning에 반영하는 루프를 한 흐름으로 학습한다.
- 내 연구와의 연결:
  - ToRA는 PAL/PoT 이후 tool-executed reasoning path의 강화판이며, 우리 path family universe에서 `RUN_TORA_LIKE`(tool-interleaved path)로 자리한다.
  - 다만 ToRA의 통제 단위는 path 내부의 reasoning↔tool interleaving이고, 내 연구는 그런 path family를 언제 추가 획득/중단할지의 상위 orchestration이다.

## 4. 핵심 방법

- reasoning path:
  - 한 해답이 `rationale → program → output → rationale → …` 형태로 자연어와 코드가 교차하는 **tool-integrated trajectory**다.
  - PAL/PoT가 보통 "한 번 program 생성 후 실행"이라면, ToRA는 실행 결과를 보고 다시 추론을 이어가는 **multi-round interleaving**이다.
- training data:
  - GSM8K·MATH 질문에 대해 GPT-4로 tool-use interactive trajectory를 수집(imitation learning source).
- imitation learning:
  - 수집한 trajectory로 LLaMA/CodeLLaMA(7B~70B)를 SFT한다.
- output space shaping:
  - imitation만으로는 다양성·교정력이 부족하므로, 모델이 만든 valid trajectory를 추가로 활용해(교정·다양화) 출력 분포를 넓힌다(rejection-sampling-like refinement).
- selection unit 관점:
  - 통제 단위는 **trajectory(path) 내부의 step·tool-call**이다.
  - heterogeneous path family 사이의 선택이나 budget allocation, STOP은 다루지 않는다.
- test-time compute:
  - 핵심은 단일 tool-integrated greedy/소수 sample이며, 명시적 adaptive TTC·verifier·search는 주방법이 아니다.

## 5. SLM-Math 관점의 재해석

- ToRA는 내 연구의 직접 경쟁자라기보다, **tool-interleaved reasoning path family의 강한 baseline이자 training-side reference**다.
- 공통점:
  - 작은 모델의 수학 reasoning을 중시한다.
  - 계산을 외부 도구로 offload하는 tool-use path의 가치를 정면으로 다룬다.
- 차이점:
  - ToRA의 핵심은 tool-interleaved trajectory를 학습으로 모델에 내재화하는 것이다.
  - 내 연구의 핵심은 (ToRA-like path를 포함한) heterogeneous path family를 제한된 TTC 안에서 state-conditioned하게 구성·선택·중단하는 것이다.
  - ToRA는 path 내부 interleaving(미시), 내 연구는 path 간 allocation·STOP(거시).
- 안전한 한 문장 차별화:
  - ToRA는 tool과 reasoning을 한 path 안에서 엮는 강한 path family를 제공하고, 내 연구는 **그 path family를 다른 family(CoT/CG/direct/verifier-guided)와 함께 언제 더 호출하고 멈출지**를 다룬다.

## 6. 우리 연구에 대한 novelty risk

- `tool-use reasoning`, `program execution으로 계산을 offload` 자체는 PAL/PoT/ToRA로 이미 강하게 선점되었다.
- `자연어 reasoning과 tool을 결합하면 좋다`도 ToRA가 이미 대표적으로 보였다.
- `tool-integrated trajectory를 학습한다`도 새롭지 않다.
- `작은 모델이 도구로 MATH를 푼다`도 ToRA가 선점.
- 따라서 위험한 framing:
  - `tool을 섞으면 math가 좋아진다`를 관찰/기여처럼 쓰기
  - `reasoning↔execution interleaving`을 우리 novelty로 쓰기
  - tool-use path 하나의 raw accuracy를 contribution으로 쓰기
- 단, ToRA는 단일 path family 강화(미시)라서, RASC/Scaling-TTC 같은 inference-time allocation 직접 baseline과는 위협 성격이 다르다. 우리 위험은 주로 "tool-use를 novelty로 오인하는 wording"과 "tool execution cost를 budget에서 누락"하는 데 있다.

## 7. 우리가 빌릴 수 있는 것

- problem framing:
  - rationale 기반 추론과 program 기반 계산은 상호보완적이다(한쪽만으로는 부족).
  - tool-integrated path는 generation cost뿐 아니라 **execution cost**도 든다.
- baseline:
  - ToRA / ToRA-Code 7B·13B·34B·70B를 **tool-interleaved path family의 strong baseline**으로 사용.
  - CoT, PAL, PoT, rejection sampling과 같은 축에 올린 비교(이미 ToRA가 제시).
- metric:
  - GSM8K·MATH accuracy + (가능하면) tool-call 횟수/실행 비용을 함께 기록 → budget-matched 비교.
- ablation:
  - `CoT only / PoT·PAL only / ToRA-like interleaved / mixed pool`을 path-family ablation에 포함.
  - output space shaping(다양화) 유무가 pool 다양성·정답률에 주는 영향 → 우리 path 다양화 논의의 reference.
  - tool execution cost를 TTC budget에 포함했을 때와 무시했을 때의 frontier 변화.
- terminology(차용):
  - `tool-integrated reasoning`, `interleaved rationale-program`, `output space shaping`, `tool-use trajectory`, `execution feedback`.

## 8. 우리가 하면 안 되는 주장

- `tool-use / program-executed reasoning을 우리가 처음 도입한다`고 쓰면 안 된다.
- `자연어 reasoning과 tool 실행을 결합하는 것이 새롭다`고 쓰면 안 된다.
- `reasoning↔execution interleaving`을 우리 contribution으로 쓰면 안 된다.
- `tool을 쓰면 math 성능이 오른다`를 새 관찰처럼 쓰면 안 된다.
- `tool-integrated path가 CoT보다 항상 우월하다`고 단정하면 안 된다.
- ToRA 대비 raw MATH/GSM8K SOTA 경쟁 프레임으로 가면 안 된다.

## 9. baseline / ablation 반영 아이디어

- path family universe에 `RUN_TORA_LIKE`(tool-interleaved) 후보를 PAL/PoT와 구분해 추가할지 검토.
- `direct / CoT / prompt-diverse CoT / PoT / PAL / ToRA-like / CG / verifier-guided`를 동일 budget 곡선에 올려 어떤 family가 어떤 문제군에서 유리한지 분석.
- **tool execution cost를 TTC budget에 명시적으로 산입**(generation token + tool call/exec). s1·DeepSeekMath 노트의 token-budget 비교와 정합.
- output space shaping이 만든 다양성을 "우리 path-pool 다양화"와 분리해, 다양성의 출처(학습 vs orchestration)를 명확히.
- ToRA-Code 계열을 strong tool-path backbone baseline으로 등록(단, 직접 재학습은 미루기).

## 10. Related Work에 넣을 문장 초안

ToRA(Gou et al., ICLR 2024)는 자연어 reasoning과 program 실행·tool 호출을 하나의 interactive trajectory 안에서 교차시키는 tool-integrated reasoning agent로, GSM8K·MATH 등에서 수집한 GPT-4 tool-use trajectory에 대한 imitation learning과 output space shaping을 통해 7B~70B 규모의 open model이 복잡한 수학 문제를 풀도록 학습한다. 이 연구는 rationale 기반 CoT의 의미적 추론과 program 기반 PAL/PoT의 정확한 계산을 한 흐름으로 결합하여, 단일 reasoning path family로는 어렵던 high-difficulty 수학 문제에서 큰 향상을 보였다.

그러나 ToRA의 초점은 tool과 reasoning이 교차하는 단일 path family를 학습으로 모델에 내재화하는 데 있으며, 제한된 test-time compute 안에서 서로 다른 reasoning path family를 어떻게 동적으로 구성하고 선택하며 언제 멈출지는 다루지 않는다. 본 연구는 ToRA-style tool-interleaved path를 heterogeneous reasoning path pool의 강한 한 family로 수용하되, 주어진 small language model이 현재 reasoning state, answer distribution, 남은 budget(생성 비용과 tool 실행 비용을 포함)을 바탕으로 어떤 path family를 추가로 획득하고 언제 멈출지를 결정하는 **state-conditioned test-time compute allocation** 문제에 위치한다.

## 11. 현재 우선순위와 다음 액션

- 현재 우선순위: `P0`
- 위협도: 중간 (tool-use novelty wording과 tool-cost budgeting을 제약하지만, inference-time allocation 직접 baseline은 아님)
- 지금 당장 해야 할 일:
  - ToRA를 PAL/PoT와 함께 **tool-executed path family cluster**의 P0 reference로 고정
  - path universe에 `ToRA-like interleaved` 후보 포함 여부 결정
  - TTC budget 정의에 **tool execution cost**를 명시적으로 추가
  - claim에서 `tool-use / reasoning-execution interleaving`을 우리 novelty로 쓰지 않도록 wording 가드
- 나중으로 미뤄도 되는 일:
  - ToRA / ToRA-Code 직접 재학습(imitation learning + output space shaping) 재현
  - GPT-4 tool-use trajectory 수집 파이프라인 재현
  - 10개 benchmark full 평가
- 한 문장 결론:
  - ToRA는 우리가 재학습할 대상이 아니라, **tool-interleaved reasoning을 heterogeneous path pool의 강한 한 family로 편입시키고 tool 실행 비용을 TTC budget에 포함하게 만드는 P0 tool-use reference**다.
