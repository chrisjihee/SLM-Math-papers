# PAL: Program-aided Language Models review

## 1. Metadata

- Title: PAL: Program-aided Language Models
- Authors: Luyu Gao, Aman Madaan, Shuyan Zhou, Uri Alon, Pengfei Liu, Yiming Yang, Jamie Callan, Graham Neubig
- Year: 2023
- Venue / Status: ICML 2023, PMLR 202
- Links:

  - Paper: https://arxiv.org/abs/2211.10435
  - Project / Code: https://reasonwithpal.com/
- Code / Data:

  - Benchmarks: `GSM8K`, `GSM-HARD`, `SVAMP`, `ASDIV`, `SINGLEEQ`, `SINGLEOP`, `ADDSUB`, `MULTIARITH`, `Colored Objects`, `Penguins`, `Date Understanding`, `Object Counting`, `Repeat Copy`
- Reading Status: `strategically-read`
- Current Priority: `P0`

## 2. One-line Summary

PAL은 LLM이 자연어 문제를 직접 계산까지 수행하게 하지 않고 Python program으로 변환하게 한 뒤, 실제 계산과 실행은 interpreter에 맡김으로써 CoT의 산술·기호 추론 오류를 줄이는 tool-use reasoning path를 제안한다.

## 3. 핵심 문제 설정

- CoT는 문제를 단계별로 분해하는 능력은 좋지만, arithmetic, state tracking, date calculation, symbolic execution에서 취약할 수 있다.
- PAL은 이 약점을 `LLM은 programmatic steps를 생성하고, 계산은 interpreter가 수행한다`는 방식으로 풀려고 한다.
- 따라서 이 논문은 `더 좋은 prompt`보다 **더 다른 reasoning path family**를 제안하는 논문이다.
- 내 연구와의 연결은 매우 직접적이다.
  - PAL은 adaptive TTC 논문은 아니지만
  - 내 연구의 heterogeneous path pool에 반드시 포함해야 할 강한 path family다.
- 즉, `RUN_COT`, `RUN_CG`, `RUN_DIRECT`만으로는 부족하고, `RUN_PAL` 또는 `RUN_POT` 같은 tool-executed path가 action space에 들어와야 한다.

## 4. 핵심 방법

- PAL의 reasoning path는 자연어 CoT가 아니라 **programmatic rationale**이다.
- LLM은 자연어 설명과 Python code가 섞인 intermediate representation을 생성하고, 최종 답은 interpreter 실행 결과로 얻는다.
- 기본 실험은 greedy decoding이며, 추가로 `k=40` majority voting도 수행한다.
- 중요한 구조적 특징:
  - reasoning decomposition은 LLM이 담당
  - arithmetic / symbolic execution은 Python interpreter가 담당
- verifier나 reward model은 없다.
  - interpreter는 verifier가 아니라 **deterministic executor**다.
- search / MCTS / reflection도 없다.
  - prompt-based program generation + execution이다.
- 모든 문제에 PAL을 적용하며, problem-dependent routing은 하지 않는다.

## 5. SLM-Math 관점의 재해석

- PAL은 내 연구에서 `tool-use path family`의 canonical reference다.
- 공통점:
  - CoT와 다른 성격의 reasoning path를 제시한다.
  - multi-sample majority voting과 결합 가능하다.
  - 일부 problem type에서 CoT보다 훨씬 높은 marginal utility를 가진다.
- 차이점:
  - PAL은 route selection이나 adaptive TTC를 다루지 않는다.
  - 내 연구는 `CoT`, `CG`, `Direct`, `PAL/PoT`, `verifier-guided path` 중 무엇을 언제 더 생성할지 고르는 orchestration 문제를 다뤄야 한다.
- 안전한 한 문장 차별화:
  - PAL은 강한 programmatic reasoning path를 제시하고, 내 연구는 **PAL을 포함한 여러 path family 중 어느 경로를 현재 문제와 state에서 더 획득할지**를 다룬다.

## 6. 우리 연구에 대한 novelty risk

- program-aided / tool-aided reasoning path 자체는 새롭지 않다.
- CoT의 arithmetic failure를 지적하는 것도 새롭지 않다.
- structured intermediate representation이 유리할 수 있다는 주장만으로는 약하다.
- tool route를 baseline에서 빼면 reviewer가 즉시 지적할 가능성이 높다.
- TTC 관점에서도 일부 문제에서는 CoT 여러 표보다 PAL 한 표가 더 cost-effective할 수 있다.
- 따라서 위험한 claim:
  - `우리가 처음으로 Python interpreter를 reasoning에 썼다`
  - `natural language reasoning과 symbolic execution 결합이 새롭다`
  - `structured rationale이 항상 CoT보다 낫다`
  - `tool-use route는 baseline이라 제외해도 된다`
- 안전한 방향:
  - PAL/PoT를 강한 path family로 받아들이고
  - **언제 PAL을 쓰고 언제 CoT/CG를 쓸지**를 allocation 문제로 다루기

## 7. 우리가 빌릴 수 있는 것

- problem framing:
  - LLM은 문제 분해에는 강하지만 계산/실행/상태 추적에는 약하다
  - reasoning path는 natural language CoT만이 아니라 programmatic rationale과 external execution을 포함한다
- baseline:
  - Direct
  - CoT
  - PAL / PoT
  - SC-CoT
  - SC-PAL
  - Mixed CoT + PAL
  - Least-to-Most + PAL
  - PAL without interpreter
- metric:
  - solve rate
  - majority@k
  - budget-accuracy curve
  - runtime error rate
  - code parse success rate
  - execution success rate
  - cost-normalized accuracy
- ablation:
  - CoT vs PAL
  - PAL vs LLM-simulated runtime
  - meaningful variable names vs random names
  - single sample vs multi-sample
  - code model vs text model
- figure/table idea:
  - route family x problem type 성능표
  - large-number robustness 비교
  - executor failure taxonomy
  - path family selection heatmap

## 8. 우리가 하면 안 되는 주장

- `우리가 처음으로 LLM 수학 추론에 Python interpreter를 사용했다`고 쓰면 안 된다.
- `우리가 처음으로 natural language reasoning과 symbolic execution을 결합했다`고 쓰면 안 된다.
- `CoT의 계산 오류 문제를 처음 지적했다`고 쓰면 안 된다.
- `structured rationale은 항상 natural language CoT보다 낫다`고 쓰면 안 된다.
- `CG는 PAL/PoT보다 일반적으로 우월하다`고 쓰면 안 된다.
- `계산 문제에서 routing은 필요 없고 CoT/CG만으로 충분하다`고 쓰면 안 된다.
- `code path의 성능은 code prompt 때문이지 execution 때문은 아니다`고 쓰면 안 된다.

## 9. baseline / ablation 반영 아이디어

- `RUN_PAL` 또는 `RUN_POT`를 action space에 공식 추가한다.
- `SC-PAL` baseline을 넣는다.
- `Mixed CoT + PAL`과 `Mixed CoT + CG + PAL`을 비교한다.
- PAL이 유리한 problem subset을 정의한다.
  - large-number
  - arithmetic-heavy
  - date / counting / state-tracking
- route-card에 PAL의 strengths, failure modes, cost를 추가한다.
- `PAL without interpreter` 또는 `LLM simulating runtime` ablation을 둔다.
- budget-accuracy curve에서 CoT / PAL / Mixed를 함께 본다.

## 10. Related Work에 넣을 문장 초안

PAL: Program-aided Language Models는 LLM이 자연어 문제를 programmatic reasoning steps로 변환하고, 실제 계산과 실행은 Python interpreter에 위임하는 방식으로 CoT의 산술 및 기호 추론 오류를 줄였다. 이 연구는 `GSM8K`, `GSM-HARD`, `SVAMP`, `ASDIV` 및 symbolic / algorithmic task에서 CoT보다 높은 성능을 보였으며, 특히 large-number arithmetic과 상태 추적 문제에서 tool-executed reasoning path의 강점을 보여주었다. 그러나 PAL은 모든 문제에 동일한 program-aided path를 적용하는 방식이며, 제한된 test-time budget 안에서 `CoT`, `PAL`, `structured rationale`, `direct solving`, `verifier-guided path` 중 어떤 reasoning path를 언제 추가로 생성하고 언제 멈출지는 다루지 않는다. 따라서 본 연구는 PAL을 하나의 강한 path family로 포함하되, 현재 reasoning state와 남은 budget을 조건으로 heterogeneous reasoning path pool을 동적으로 구성하는 문제에 초점을 둔다.

## 11. 현재 우선순위와 다음 액션

- 현재 우선순위: `P0`
- 위협도: 중간~높음
- 지금 당장 해야 할 일:
  - `PAL / PoT`를 path family 목록에 공식 추가
  - macro strategy universe에 `PAL-only`, `CoT+PAL`, `CoT trio + PAL`, `PAL after disagreement`를 추가
  - `SC-CoT`, `SC-PAL`, `SC-mixed`를 budget-matched baseline으로 설계
  - route-card에 PAL의 cost, strengths, failure modes를 작성
- 나중으로 미뤄도 되는 일:
  - PAL의 모든 symbolic benchmark 재현
  - `GSM-HARD` full regeneration
  - Python 이외 solver 확장
- 한 문장 결론:
  - PAL은 경쟁자가 아니라, **내 method가 반드시 선택·조합해야 하는 강한 candidate path family**다.
